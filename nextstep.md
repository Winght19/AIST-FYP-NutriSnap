Phase 1: Define and Finalize Your Data Contracts
Before writing a single line of application logic, you need absolute clarity on what data lives where and what it looks like in each system. This phase is about design decisions, not code.

Step 1 — Audit Your Existing SwiftData Models
Go through your existing User, Meal, FoodLog, and Food models and identify every field. For each field, ask three questions: Does this field need to be synced to the cloud? Does this field need to survive a full app deletion and re-install? Does this field need to be available offline? Any field that answers yes to any of these belongs in the sync layer. Fields that are purely cosmetic or ephemeral (like a UI selection state) can stay local-only and should be marked as such in your mental model.

Step 2 — Add Sync Metadata to Every Syncable Model
Every model that travels between SwiftData and the cloud needs three additional fields added to it. The first is a remoteID, which is the cloud's authoritative identifier for that record. This is separate from SwiftData's internal persistentModelID. The remoteID starts as nil for records created locally before they've been pushed to the cloud, and gets populated on the first successful cloud write. The second is a needsSync boolean flag, which is your dirty marker. It starts as true on any local create or update, and flips to false only after the cloud confirms a successful write. This is your recovery mechanism for offline scenarios and network failures. The third is a lastModifiedAt timestamp, which is your conflict resolution signal. When the same record exists both locally and in the cloud, whichever version has the later timestamp wins. This simple "last write wins" strategy is appropriate for a health app where a single user owns all their data.

Step 3 — Define the isProfileComplete Flag on the User Model
This single boolean on your User model is what drives the entire routing decision between OnboardingView and DashboardView for existing users. When a user is first created from a Google Sign-In, this flag is false. After they finish onboarding and the profile is saved to the cloud, the backend sets it to true and returns it in every subsequent profile fetch. Your routing logic will inspect this flag on every app launch, not just the first one. This means if a user somehow abandoned onboarding mid-way and force-quit the app, they will be correctly routed back to finish it rather than being dropped into a half-configured dashboard.

Step 4 — Design Your Cloud Data Model
On your backend (whether that is Firebase, Supabase, a custom Node/Python API, or otherwise), your database schema needs to mirror the structure of your SwiftData models with one critical addition: every FoodLog and Meal record must carry a userId foreign key that points to the owning User. This is not optional — it is the security boundary that prevents one user's data from ever being accessible to another. Your User table needs to store the Google sub claim as its primary identifier, since this is the stable, unique, and immutable identifier Google provides across sign-ins. Your backend should compute and store the personalized calorie and macro targets after onboarding, because that computation logic belongs on the server, not in the client.

Step 5 — Define Your DTO Layer
Create a separate group in your Xcode project called DTOs or NetworkModels. These are pure Codable Swift structs whose only job is to represent the shape of data as it crosses the network boundary. They have zero relationship to SwiftData. You need DTOs for the following scenarios: the auth request sent to your backend containing the Google ID token, the auth response from your backend containing your app's JWT and a isNewUser flag, the full UserProfileDTO used for fetching and returning profile data, the OnboardingProfileDTO containing the fields collected during onboarding, and the FoodLogDTO and MealDTO for sync operations. The strict separation between your DTOs and your SwiftData models is the most important architectural boundary in this entire system. Every conversion between them should happen in one explicit, dedicated mapping function, never scattered across views.

Phase 2: Structure Your Service Layer
Your service layer is the group of classes that handle communication with the outside world. Nothing in your SwiftUI views should know about URLs, JSON decoding, or Keychain operations directly.

Step 6 — Build Your APIClient
Create a single APIClient class responsible for constructing and executing all network requests. It handles three responsibilities: attaching the Bearer token from Keychain to every outbound request's Authorization header, encoding request bodies from DTOs using JSONEncoder, and decoding responses back into DTOs using JSONDecoder. Both the encoder and decoder must have their date strategy set to .iso8601 to ensure consistent date serialization across platforms. The APIClient should throw a specific unauthenticated error if it is asked to make a request but finds no token in Keychain, which will trigger a sign-out in the layer above.

Step 7 — Build Your KeychainManager
This is a small utility that wraps the Security framework's SecItem functions. It needs exactly three operations: save a token string, retrieve a token string, and delete a token. The token being stored here is your app's own JWT returned from your backend after the Google Sign-In exchange — not the Google ID token itself, which is short-lived and should not be stored. The Keychain is the only acceptable place to store this credential. UserDefaults is not encrypted and must not be used for tokens.

Step 8 — Build Your UserProfileService
This service wraps the specific API calls related to user identity. It needs four methods: authenticate(idToken:) which sends the Google token to your backend and returns the AuthResponseDTO, fetchProfile(token:) which retrieves an existing user's full profile using a stored JWT, saveOnboardingProfile(_:forUserID:) which sends the onboarding form data to the backend and receives back the computed profile including calorie targets, and updateProfile(_:forUserID:) for future profile edits. This service takes the raw UserProfileDTO from the APIClient and returns it — it does not touch SwiftData. That boundary is always respected.

Step 9 — Build Your SyncService
This service handles the bidirectional flow of FoodLog and Meal data. It has three distinct responsibilities. First, the initial sync that runs when an existing user logs in — this pulls all of their cloud records into SwiftData and flushes any locally-dirty records back to the cloud. Second, the write-through sync that runs immediately after every local create or update — this attempts a cloud write in the background while the UI is already showing the local change. Third, the reconciliation flush that runs every time the app returns to the foreground from background — this finds every record where needsSync == true and retries the cloud write. The SyncService is the only place where the upsert-with-conflict-resolution logic lives. The "last modified timestamp wins" comparison happens here before any SwiftData record is overwritten with cloud data.

Phase 3: Design the App State Machine
This is the architectural heart of your routing system. Getting this right means every screen transition in the app is predictable, testable, and driven from one place.

Step 10 — Define Your AppState Enum
Create an AppState enum with the following cases. launching represents the moment between app open and the first asynchronous check completing — this is when you show your splash screen. unauthenticated means no valid session exists and LandingView should be shown. onboarding means the user is authenticated but isProfileComplete is false and OnboardingView should be shown. syncing means the user is authenticated and complete, but the initial data pull from the cloud is still in progress — this prevents the dashboard from rendering with stale or empty data. authenticated means everything is ready and DashboardView should be shown. error carries an associated string message and represents a non-fatal failure with a recovery path back to unauthenticated. These six cases represent every possible state the app can be in, and no screen transition happens without going through this enum.

Step 11 — Build Your AppStateManager
Create an @Observable class marked @MainActor called AppStateManager. This is the single object that owns appState and is responsible for every transition between states. It holds a reference to your service layer objects. It exposes one method for each user-initiated or system-initiated event: handleAppLaunch(modelContext:), handleGoogleSignIn(modelContext:), completeOnboarding(with:modelContext:), and signOut(modelContext:). Each of these methods follows the same pattern: perform async work, then set appState to the appropriate next case based on the result. The @MainActor annotation is not optional here — it ensures all appState mutations happen on the main thread, which is required for SwiftUI to respond to them correctly, and also satisfies SwiftData's requirement that ModelContext operations happen on the main thread.

The handleAppLaunch method checks Keychain for a stored token. If none exists, it immediately transitions to .unauthenticated. If a token exists, it calls UserProfileService.fetchProfile(token:) to validate the session against the backend. If the backend accepts the token and returns a profile, it runs the sync flow and transitions to .authenticated. If the backend rejects the token (expired, revoked), it clears the Keychain entry and transitions to .unauthenticated. The user sees the splash screen for the duration of this check.

The handleGoogleSignIn method calls AuthenticationManager to trigger the Google SDK flow and get the ID token. It then calls UserProfileService.authenticate(idToken:). The backend response tells it whether the user is new or existing via isNewUser. New users get a minimal stub User record inserted into SwiftData and the state transitions to .onboarding. Existing users go through the sync flow and transition to .authenticated.

Step 12 — Define the Upsert Pattern for User Records
Within AppStateManager, write a dedicated private helper called upsertUser that takes a UserProfileDTO and a ModelContext. This function uses a FetchDescriptor with a predicate matching remoteID to check whether a User record already exists locally. If it does, it updates the existing record's fields from the DTO. If it does not, it creates a new User record and inserts it. The actual field-mapping logic should live in a separate private apply(dto:to:) function that is called by both the create and update paths. This ensures the mapping logic never duplicates. This upsert pattern is used both on login and on every subsequent profile fetch.

Phase 4: Wire Up the Routing in NutriSnapApp.swift
Step 13 — Instantiate AppStateManager at the Root
In NutriSnapApp.swift, declare AppStateManager as a @State property. This is the correct wrapper because @Observable objects owned by a Scene or App should use @State to ensure SwiftUI manages their lifetime correctly. Pass it into the view hierarchy using .environment() on your root view. Because you're using the @Observable macro rather than ObservableObject, you do not use .environmentObject() — you use the plain .environment() modifier. Any view in the entire hierarchy can then access it with @Environment(AppStateManager.self).

Step 14 — Build RootView as a Pure Router
RootView should contain no user-facing UI of its own. It is a switch statement on appStateManager.appState that renders the appropriate top-level view for each case. Wrap the switch in a ZStack and apply an .animation modifier keyed on appState to get smooth crossfade transitions between screens without any manual animation code in your child views. Attach .task to RootView to call appStateManager.handleAppLaunch(modelContext:). The .task modifier is the correct choice over .onAppear here because it properly ties the async work to the view's lifetime and cancels it if the view disappears, which prevents duplicate launches.

Step 15 — Keep Child Views Thin and Decoupled
LandingView should only know how to trigger handleGoogleSignIn via the environment and display sign-in UI. It should not contain any routing logic of its own. OnboardingView should collect form data in local @State properties across its steps and call completeOnboarding(with:modelContext:) on the final step submission. It does not decide what happens after that — the state machine does. DashboardView receives currentUser via the environment and should use @Query with a predicate filtered to that user's remoteID for all its data needs. The dashboard never fetches data from the cloud directly — it always reads from SwiftData and trusts the sync layer to keep it fresh.

Phase 5: Implement the Sync Strategy
Step 16 — Adopt the Write-Through with Dirty-Flag Pattern
Every time the user creates or modifies a FoodLog or Meal, the sequence is always: write to SwiftData first with needsSync = true, save the context immediately so @Query updates and the UI reflects the change instantly, then fire an async Task to attempt the cloud write in the background. If the cloud write succeeds, clear needsSync and store the returned remoteID. If it fails for any reason — network timeout, server error, device offline — do nothing beyond ensuring needsSync remains true. The record will be picked up by the next reconciliation pass. The user experiences zero latency and the data is never lost.

Step 17 — Handle the Initial Sync Carefully
When an existing user logs in, you need to pull their cloud data before showing the dashboard, or you risk showing them an empty screen while sync catches up. This is why the .syncing state exists. During this state, show a loading indicator or skeleton screen. The initial sync pulls FoodLog and Meal records for the past 30 days (or whatever window makes sense for your dashboard) rather than the entire history, which keeps the login experience fast. Older records can be pulled lazily when the user navigates to a history view. After pulling, run flushPendingSync to push up any locally-dirty records that may have accumulated since the last session (offline edits, failed writes). Only after both operations complete should you transition to .authenticated.

Step 18 — Use scenePhase for Foreground Reconciliation
In your DashboardView, observe .scenePhase from the environment. When the phase transitions to .active, trigger SyncService.flushPendingSync followed by a lightweight cloud pull for recent records. This is your background-resilience mechanism. If the user makes changes on another device, closes the app, and reopens it, this foreground hook ensures they get the latest data within seconds. It also catches any writes that failed due to a spotty connection while the app was last in use.

Step 19 — Implement Conflict Resolution with Last-Write-Wins
In SyncService, when pulling records from the cloud and comparing them to local SwiftData records with the same remoteID, always compare lastModifiedAt timestamps before overwriting. If the cloud record is newer, overwrite the local record and set needsSync = false. If the local record is newer or equal, skip the cloud version — your local copy is the most recent truth and the next flushPendingSync will push it up. This avoids a race condition where a foreground sync pull would overwrite a local edit the user just made but that hasn't been pushed yet.

Phase 6: Security and Relationship Integrity
Step 20 — Enforce Cascade Deletes in SwiftData Relationships
On the User model, the relationship to FoodLog and Meal must have deleteRule: .cascade and correctly declared inverse: parameters pointing back to the owner property on each child model. This ensures that if a user's local record is deleted (on sign-out, for example), all of their associated logs and meals are deleted with it, leaving no orphaned records in SwiftData. The inverse: parameter is what makes @Query predicates like "give me all FoodLogs belonging to this user" work correctly, because SwiftData uses it to maintain the bidirectional pointer between parent and child records.

Step 21 — Secure the Cloud Relationships with JWT Validation
On the backend, every endpoint that reads or writes FoodLog or Meal data must validate that the userId in the request matches the sub claim in the incoming JWT. This cannot be trusted to the client. It must be enforced server-side on every single request. A user should never be able to pass another user's ID to your endpoints and receive or modify that user's data, regardless of what the iOS app sends. This backend validation is the security guarantee that makes the entire relationship model trustworthy.

Step 22 — Handle Sign-Out Completely
When signOut is called on AppStateManager, three things must happen in order: the Google SDK session must be cleared via AuthenticationManager, the JWT must be deleted from Keychain, and currentUser must be set to nil. You also need to decide whether to delete local SwiftData records on sign-out or leave them on device. Leaving them is more resilient for users who temporarily sign out and back in, but deleting them is safer on shared devices. The right default for a personal health app is to leave the SwiftData records on device and reload them from the cloud on the next sign-in. This makes re-login feel instant.
